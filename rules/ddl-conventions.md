# DDL Conventions

**Single responsibility:** Conventions for schema objects created via DDL —
tables, columns, constraints, triggers, seed data. Query/SQL formatting lives
in `sql-format.md`; package/procedure standards live in `plsql-standards.md`.

> Consolidated from `oracle-apex_standards.mdc` §2.2 (constraint naming) and
> §3 (database/DDL), `docs/coding-standards/plsql/{naming,tables}.md`,
> `.cursor/rules/table-ddl.mdc`, and the seed-data pattern pulled forward from
> `tmp/templates.md` §4 before archiving it. Originals archived under
> `_archive/coding-standards-legacy/`.

## 1. Object & Constraint Naming

- All object names in `snake_case`.
- **Project prefix**: all core architectural objects (tables, views, packages) begin with the approved project prefix (e.g. `prefix_`).
  - Tables: plural (e.g. `prefix_tickets`)
  - Views: plural + `_vw` suffix (e.g. `prefix_tickets_vw`)
- **Boolean flags**: columns storing a `'Y'`/`'N'` value use a `_yn` suffix (e.g. `active_yn`, `is_custom_yn`). Do not use `_yn` for numeric or date columns. See `plsql-standards.md` §1 for the equivalent convention on PL/SQL parameters and variables (e.g. `p_require_mfa_yn`).
- **Constraints** always use prefixes:

| Prefix | Type | Pattern |
|---|---|---|
| `fk_` | Foreign Key | `fk_<table>_<ref_table>` |
| `uk_` | Unique | `uk_<table>_<column>` |
| `ck_` | Check | `ck_<table>_<description>` |

**GOOD**: `alter table prefix_tickets add constraint fk_tickets_users foreign key (user_id) references users (user_id);`
**BAD**: `alter table prefix_tickets add constraint tickets_users_fk foreign key (user_id) references users (user_id);`

## 2. Primary Key

The PK column follows `<entity_name>_id` — the **entity name without the module prefix**. The module prefix belongs on the table name for schema-level uniqueness; columns use only the entity name for readability. Use identity columns for auto-generation.

**GOOD**:
```sql
-- Table prefix is "prefix_", entity is "ticket" → PK is "ticket_id"
create table prefix_tickets (
    ticket_id number generated by default on null as identity (start with 1) primary key not null
  , description varchar2(200 char)
);
```

**BAD**: a generic `id` column, or repeating the table prefix inside the PK column name.

## 3. Data Types & Char Semantics

- **Strings**: always `char` byte semantics (`varchar2(100 char)`) — never `byte` or implicit size, in both `create table` and `alter table`.
- **Numbers**: `number` for IDs, `number(p,s)` for decimals.
- **Dates**: `date` when time precision isn't needed.
- **Timestamps**: `timestamp with local time zone` for audit events.
- **CLOB**: large text fields (descriptions, notes).

**BAD**: `varchar2(100)` or `varchar2(255 byte)` — silently truncates multi-byte characters.

## 4. Standard Audit Columns & Active Flag

Every table includes the standard active flag and audit columns, coalescing `sys_context` so both APEX sessions and direct DB connections are supported:

```sql
  , active_yn                    varchar2(1 char) default 'Y' not null
  , created_by                   varchar2(60 char) default
                                  coalesce(
                                    sys_context('APEX$SESSION','app_user')
                                  , regexp_substr(sys_context('userenv','client_identifier'),'^[^:]*')
                                  , sys_context('userenv','session_user')
                                  )
                                  not null
  , created_on                   timestamp with local time zone default localtimestamp not null
  , last_updated_by              varchar2(60 char)
  , last_updated_on              timestamp with local time zone
  , constraint ck_prefix_tickets_active_yn check (active_yn in ('Y', 'N'))
```

## 5. Trigger Patterns (Audit Automation)

Maintain audit columns automatically using a **compound trigger**:

```sql
create or replace trigger prefix_tickets_compound_trg
for insert or update on prefix_tickets
compound trigger

  before each row is
  begin
    if updating then
      :new.last_updated_on := localtimestamp;
      :new.last_updated_by := coalesce(
                                sys_context('APEX$SESSION','app_user')
                              , regexp_substr(sys_context('userenv','client_identifier'),'^[^:]*')
                              , sys_context('userenv','session_user')
                              );
    end if;
  end before each row;

end prefix_tickets_compound_trg;
/
```

> **Soft-deletes**: when performing `update ... set active_yn = 'N'`, do **not** set `last_updated_by`/`last_updated_on` in application code — the compound trigger handles it automatically.

## 6. Table Comments

Comments are mandatory via `execute immediate` so scripts are re-runnable:

```sql
begin
  execute immediate 'comment on table prefix_tickets is ''Tracks all support tickets in the system.''';
  execute immediate 'comment on column prefix_tickets.ticket_id is ''Unique identifier for the ticket.''';
end;
/
```

## 7. DDL File Structure

Order for a table's SQL file:

1. **Table creation** (with inline constraints)
2. **Performance indexes**
3. **Triggers** (compound trigger for audit)
4. **Comments** (grouped in a `begin...end;` block)

See [`table_template.sql`](../templates/table_template.sql) and [`compound_trigger_template.sql`](../templates/compound_trigger_template.sql) for ready-to-copy skeletons that already follow this order.

## 8. Seed Data / Lookup Table Population

Data scripts live in the project's `data/` folder (see `repo-structure.md`) and are re-runnable. For LOV/lookup tables, load a JSON payload and `merge` it in rather than writing one-off `insert` scripts by hand — this keeps data changes reviewable as a diff and re-runnable on any environment.

```sql
set define off;

prompt CHANGE_ME data

declare
  l_json clob;
begin
  l_json := q'!
[
  {
    "CHANGE_ME_code": "CHANGEME",
    "CHANGE_ME_name": "CHANGEME",
    "CHANGE_ME_seq": 1
  }
]
!';

  for data in (
    select *
      from json_table(l_json, '$[*]' columns
        change_me_code varchar2(4000 char) path '$.CHANGE_ME_code'
      , change_me_name varchar2(4000 char) path '$.CHANGE_ME_name'
      , change_me_seq  number             path '$.CHANGE_ME_seq'
      )
  )
  loop
    -- Looping over each entry makes debug failures easier to track
    merge into change_me dest
    using (
      select data.change_me_code as change_me_code
        from dual
    ) src
    on (dest.change_me_code = src.change_me_code)
    when matched then
      update
         set dest.change_me_name = data.change_me_name
           , dest.change_me_seq  = data.change_me_seq
    when not matched then
      insert (
             change_me_code
           , change_me_name
           , change_me_seq
      )
      values (
             data.change_me_code
           , data.change_me_name
           , data.change_me_seq
      );
  end loop;
end;
/
```

> See `sql-format.md` §9 for the general `merge`/seed-alignment formatting rules this pattern follows.
